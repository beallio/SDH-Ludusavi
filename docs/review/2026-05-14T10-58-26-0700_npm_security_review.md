# NPM / pnpm Security Review

Review date: 2026-05-14

Scope:

- `package.json`
- `pnpm-lock.yaml`
- `node_modules` as installed by pnpm
- frontend build/config scripts that consume NPM modules

Commands run:

- `npm exec --yes pnpm@10.23.0 -- audit --json`
- `npm exec --yes pnpm@10.23.0 -- audit --prod --json`
- `npm exec --yes pnpm@10.23.0 -- list --depth Infinity --json`
- `npm exec --yes pnpm@10.23.0 -- why webpack`
- `npm exec --yes pnpm@10.23.0 -- why picomatch`
- `npm exec --yes pnpm@10.23.0 -- why minimatch`
- `npm exec --yes pnpm@10.23.0 -- why brace-expansion`
- `npm exec --yes pnpm@10.23.0 -- why serialize-javascript`
- `npm exec --yes pnpm@10.23.0 -- why ajv`

Important audit context:

- Production audit: 0 vulnerabilities across 7 production dependencies.
- Full audit: 18 vulnerabilities across 216 dependencies: 10 high, 6 moderate, 2 low, 0 critical.
- The vulnerable packages are in the build/dev graph, not the shipped runtime dependency graph.
- Build-time risk is still relevant because `dist/index.js` is packaged into the Decky plugin.

## Issue 1

* **[Severity]** High
* **File & Function/Line:** `package.json`, lines 29-46
* **Description:** `package.json` defines `dependencies` twice. JSON parsers keep the last key, so the earlier `react-router` entry is silently discarded. This creates an unreliable dependency manifest: local installs, audits, and CI can disagree with developer intent. It also explains the unresolved `react-router` type/runtime path seen during TypeScript validation.
* **Vulnerable Code:**

```json
"dependencies": {
  "react-router": "^6.0.0"
},
...
"dependencies": {
  "@decky/api": "^1.1.3",
  "react-icons": "^5.3.0",
  "tslib": "^2.7.0"
}
```

* **Proposed Fix:**

```json
"private": true,
"packageManager": "pnpm@10.23.0",
"dependencies": {
  "@decky/api": "1.1.3",
  "react-icons": "5.3.0",
  "react-router": "6.30.2",
  "tslib": "2.8.1"
}
```

Also regenerate `pnpm-lock.yaml` with pnpm and commit the resulting lockfile. If `react-router` is only needed for Decky package type declarations, prefer the smallest direct dependency that satisfies the build and document why it is present.

## Issue 2

* **[Severity]** High
* **File & Function/Line:** `pnpm-lock.yaml`, transitive dev dependency through `@types/webpack`; traced with `pnpm why webpack`
* **Description:** `@types/webpack@5.28.5` pulls `webpack@5.95.0`, which has active advisories in the audit report. The current project builds with Rollup, not webpack, so this type package appears unnecessary and is bringing in a large vulnerable toolchain solely for types.
* **Vulnerable Code:**

```json
"devDependencies": {
  "@types/webpack": "^5.28.5"
}
```

Audit trace:

```text
@types/webpack 5.28.5
└─┬ webpack 5.95.0
  └─┬ terser-webpack-plugin 5.3.10
    └── webpack 5.95.0 peer
```

* **Proposed Fix:**

```json
"devDependencies": {
  "@decky/rollup": "1.0.2",
  "@decky/ui": "4.11.0",
  "@rollup/rollup-linux-x64-musl": "4.53.3",
  "@types/react": "19.1.1",
  "@types/react-dom": "19.1.1",
  "rollup": "4.53.3",
  "typescript": "5.6.2"
}
```

Remove `@types/webpack` if no local source imports webpack types. Then run `pnpm install --lockfile-only`, `pnpm audit`, and the frontend build/type checks.

## Issue 3

* **[Severity]** High
* **File & Function/Line:** `pnpm-lock.yaml`, transitive dev dependency through `@decky/rollup`; traced with `pnpm why picomatch`
* **Description:** `@decky/rollup@1.0.2` pulls vulnerable `picomatch` versions through Rollup plugin utilities and `rollup-plugin-delete`. The high-severity advisories are ReDoS issues in glob pattern parsing. This is primarily build-time exposure, but build tooling processes local project paths and can be influenced by repo content.
* **Vulnerable Code:**

```text
@decky/rollup 1.0.2
├─┬ @rollup/plugin-commonjs 26.0.3
│ └─┬ @rollup/pluginutils 5.3.0
│   └── picomatch 4.0.3
└─┬ rollup-plugin-delete 2.2.0
  └─┬ del 6.1.1
    └─┬ globby 11.1.0
      └─┬ fast-glob 3.3.3
        └─┬ micromatch 4.0.8
          └── picomatch 2.3.1
```

* **Proposed Fix:**

```json
"pnpm": {
  "overrides": {
    "picomatch@^4.0.0": "4.0.4",
    "picomatch@^2.0.0": "2.3.2"
  }
}
```

Then run `pnpm install --lockfile-only`, `pnpm audit`, and `pnpm run build`. Prefer upgrading `@decky/rollup` once it publishes fixed transitive versions; use overrides as a contained interim mitigation.

## Issue 4

* **[Severity]** Medium
* **File & Function/Line:** `pnpm-lock.yaml`, transitive dev dependency through `@decky/rollup`; traced with `pnpm why minimatch` and `pnpm why brace-expansion`
* **Description:** The build graph contains vulnerable `minimatch` / `brace-expansion` chains. The reported impact is denial of service from crafted brace/glob patterns. This is lower risk than runtime exposure because patterns are developer-controlled in this repo, but it is still avoidable supply-chain debt in build tooling.
* **Vulnerable Code:**

```text
@decky/rollup 1.0.2
├─┬ @rollup/plugin-commonjs 26.0.3
│ └─┬ glob 10.5.0
│   └── minimatch 9.0.5
└─┬ rollup-plugin-delete 2.2.0
  └─┬ del 6.1.1
    └─┬ rimraf 3.0.2
      └─┬ glob 7.2.3
        └── minimatch 3.1.2
```

* **Proposed Fix:**

```json
"pnpm": {
  "overrides": {
    "brace-expansion@^2.0.0": "2.0.3",
    "brace-expansion@^1.0.0": "1.1.12"
  }
}
```

`brace-expansion@1.1.12` was present in the old glob chain and was not one of the patched 2.x targets from the audit output. If the audit continues to flag the old chain, the safer long-term fix is replacing or upgrading the `rollup-plugin-delete` chain through a newer `@decky/rollup`.

## Issue 5

* **[Severity]** Medium
* **File & Function/Line:** `package.json`, lines 29-46 and `pnpm-lock.yaml`
* **Description:** Most package versions use ranges (`^`). The lockfile pins the current install, but ranges make future lockfile refreshes pull new transitive code without a manifest diff. For a plugin that packages generated JavaScript, conservative pinning reduces surprise during release builds.
* **Vulnerable Code:**

```json
"@decky/api": "^1.1.3",
"react-icons": "^5.3.0",
"tslib": "^2.7.0",
"@decky/rollup": "^1.0.2",
"@decky/ui": "^4.11.0",
"rollup": "^4.53.3",
"typescript": "^5.6.2"
```

* **Proposed Fix:**

```json
"dependencies": {
  "@decky/api": "1.1.3",
  "react-icons": "5.3.0",
  "react-router": "6.30.2",
  "tslib": "2.8.1"
},
"devDependencies": {
  "@decky/rollup": "1.0.2",
  "@decky/ui": "4.11.0",
  "@rollup/rollup-linux-x64-musl": "4.53.3",
  "@types/react": "19.1.1",
  "@types/react-dom": "19.1.1",
  "rollup": "4.53.3",
  "typescript": "5.6.2"
}
```

Use an explicit upgrade workflow (`pnpm up --latest`, audit, build, test) instead of opportunistic range movement.

## Issue 6

* **[Severity]** Medium
* **File & Function/Line:** `package.json`, lines 6-10; `scripts/pre_commit.sh`, lines 18-33
* **Description:** The frontend has no security gate in package scripts or pre-commit. Python checks run on commit, but frontend dependency auditing, TypeScript checking, and Rollup build validation are not part of the enforced gate. That allows vulnerable lockfile updates or TypeScript failures to merge unnoticed.
* **Vulnerable Code:**

```json
"scripts": {
  "build": "rollup -c",
  "watch": "rollup -c -w",
  "test": "echo \"Error: no test specified\" && exit 1"
}
```

* **Proposed Fix:**

```json
"scripts": {
  "build": "rollup -c",
  "typecheck": "tsc --noEmit",
  "audit": "pnpm audit --prod && pnpm audit --audit-level high",
  "verify": "pnpm run typecheck && pnpm run build && pnpm run audit",
  "watch": "rollup -c -w"
}
```

Then add `pnpm run verify` to the local pre-commit or release/package gate once the current TypeScript and audit findings are fixed.

## Issue 7

* **[Severity]** Medium
* **File & Function/Line:** repository root, missing `.npmrc`; README lines 73-77
* **Description:** The project documents `pnpm install`, but there is no project-level npm/pnpm policy. Without it, installs may run dependency lifecycle scripts, use whatever package manager version happens to be installed, and write cache/log data outside the intended cache area.
* **Vulnerable Code:**

```bash
pnpm install
```

* **Proposed Fix:**

Add `.npmrc`:

```ini
package-manager-strict=true
strict-peer-dependencies=false
auto-install-peers=true
resolution-mode=highest
ignore-scripts=true
audit-level=moderate
fund=false
```

If a dependency genuinely needs build scripts, enable it narrowly with pnpm's `onlyBuiltDependencies` rather than allowing all lifecycle scripts.

## Issue 8

* **[Severity]** Low
* **File & Function/Line:** `scripts/post_commit.sh`, lines 19-23
* **Description:** The post-commit package path builds and then automatically copies the plugin zip to a fixed LAN host if it responds to ping. This is not an NPM package vulnerability, but it is part of the frontend release path. A compromised build dependency could produce a malicious `dist/index.js`, and the hook would then deploy it automatically to the Steam Deck without an explicit approval step.
* **Vulnerable Code:**

```bash
if ping -c 1 -W 2 10.168.168.20 >/dev/null 2>&1; then
  echo "Pushing plugin to Steam Deck..."
  scp ./out/SDH-ludusavi.zip deck@10.168.168.20:/home/deck/Downloads/
else
  echo "Steam Deck (10.168.168.20) not reachable, skipping push."
fi
```

* **Proposed Fix:**

```bash
if [[ "${SDH_LUDUSAVI_AUTO_PUSH:-}" == "1" ]]; then
  if ping -c 1 -W 2 10.168.168.20 >/dev/null 2>&1; then
    echo "Pushing plugin to Steam Deck..."
    scp ./out/SDH-ludusavi.zip deck@10.168.168.20:/home/deck/Downloads/
  else
    echo "Steam Deck (10.168.168.20) not reachable, skipping push."
  fi
else
  echo "Set SDH_LUDUSAVI_AUTO_PUSH=1 to push the plugin zip to the Steam Deck."
fi
```

## Recommended Implementation Order

1. Fix `package.json` duplicate `dependencies`, add `private: true`, and add `packageManager: "pnpm@10.23.0"`.
2. Remove `@types/webpack` if not required; this should eliminate the `webpack`, `serialize-javascript`, and `ajv` audit findings.
3. Add pnpm overrides for `picomatch` and any still-vulnerable `brace-expansion` chain that can be safely overridden.
4. Add `typecheck`, `audit`, and `verify` package scripts.
5. Add `.npmrc` with strict package-manager behavior and disabled lifecycle scripts by default.
6. Add frontend verification to the local gate once `tsc --noEmit` and audit are green.
7. Require opt-in for post-commit auto-push to the Steam Deck.

## Residual Risk

The production dependency graph currently audits clean, so the immediate shipped runtime module risk is low. The main risk is build-time supply-chain compromise or vulnerable build tooling producing a compromised `dist/index.js`. The best hardening path is to shrink the dev graph, pin versions, audit the pnpm lockfile in the release path, and make generated `dist/` reproducible from a clean install.
