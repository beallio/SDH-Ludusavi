## 🔍 The Review Finding
**Reviewer Comment / Rationale:**
> "A successful backup can be reported as a failed operation and lose its history entry if the follow-up status refresh fails. `force_backup()` and `handle_game_exit()` perform the Ludusavi backup, then call `_refresh_statuses_unlocked()`, and only afterwards record `backed_up` history. If the backup succeeds but refresh fails because Ludusavi discovery, config reads, or status parsing fail, the caller sees an exception even though the destructive/externally visible backup already happened, and durable history does not show the successful backup."

## 📍 Exact Location
- **Original PR (if applicable):** #
- **File(s):** `py_modules/sdh_ludusavi/service.py`
- **Function/Class/Lines:** `SDHLudusaviService.handle_game_exit` lines 568-578; `SDHLudusaviService.force_backup` lines 588-598

## 🛠️ Requested Change
- Record the successful backup history immediately after `_run_locked()` returns successfully, before any follow-up refresh.
- If post-backup refresh fails, do not convert the completed backup into a failed backup result. Log the refresh failure and return a successful backup result with the refresh error represented separately if needed.
- Add regression coverage for "backup succeeds, refresh fails" for both manual backup and auto-exit backup paths.

- [ ] Existing unit tests still pass (no regressions).
- [ ] Add tests proving successful backup history is persisted when `_refresh_statuses_unlocked()` fails after the backup operation.

## 🔍 The Review Finding
**Reviewer Comment / Rationale:**
> "State loading only verifies that each history slot is a dict; it does not validate the expected schema or scalar types. A corrupted or manually edited `state.json` can load `timestamp` as a non-string, omit fields, or include arbitrarily large nested objects. The backend then returns that object through `refresh_games()`, and the frontend calls `selectedHistory.timestamp.split(' ')`, which can crash the panel. This is a robustness and local-state hardening gap."

## 📍 Exact Location
- **Original PR (if applicable):** #
- **File(s):** `py_modules/sdh_ludusavi/service.py`, `src/index.tsx`
- **Function/Class/Lines:** `SDHLudusaviService._load_state` lines 713-725; `Content` render block line 579

## 🛠️ Requested Change
- Introduce a narrow coercion/validation helper for persisted operation history entries.
- Only retain known fields: `operation`, `trigger`, `status`, `reason`, `message`, and `timestamp`.
- Validate allowed enum values and require `timestamp` to be a string before exposing entries to the frontend.
- Drop or sanitize malformed entries instead of returning arbitrary dicts.

## ✅ Acceptance Criteria
- [ ] The code implements the requested change.
- [ ] Existing unit tests still pass (no regressions).
- [ ] Add tests for malformed `game_history` entries with missing timestamp, non-string timestamp, invalid status, and nested unexpected values.
- [ ] The frontend cannot crash on persisted malformed history entries returned by the backend.
